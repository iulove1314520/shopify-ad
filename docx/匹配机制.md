# 匹配机制说明（v2 — 保守型双路径规则引擎）

## 1. 文档目的

本文基于重写后的代码实现，说明系统中"访客点击记录"与"Shopify 订单"之间的匹配机制。v2 机制在 v1 基础上做了全面改造，核心变化为：

- 时间窗从 3 天收紧到 **24 小时**
- 引入**商品指纹**（token 级模糊匹配）替代原有精确相等
- 引入 **IP/Geo 分层证据**
- 引入**主路径 / 降级路径双路径决策引擎**
- 引入**访客占用模型**，同一访客同时只能归属一个订单
- 新增**人工撤销释放**能力
- 匹配结果写入 **match_mode / lead_score_gap / decision_summary**

## 3. 架构总览

新机制分为 4 层：

```
┌──────────────────────┐
│   候选筛选层          │  24h 内、有 click ID、未被占用
├──────────────────────┤
│   证据提取层          │  商品指纹 + 时间分 + IP/Geo 证据
├──────────────────────┤
│   决策层              │  主路径 / 降级路径 + 领先门槛
├──────────────────────┤
│   编排层              │  写库、更新订单、回传平台、人工撤销
└──────────────────────┘
```

## 4. 候选筛选层

### 4.1 时间窗

默认 **24 小时**（硬编码）。候选访客必须满足：

- `timestamp >= order.created_at - 24h`
- `timestamp <= order.created_at`

### 4.2 资格条件

- 至少有一个点击参数：`ttclid` 或 `fbclid`
- **未被活跃匹配占用**：通过 `NOT EXISTS (SELECT 1 FROM matches WHERE visitor_id = v.id AND active = 1)` 排除

## 5. 证据提取层

### 5.1 商品指纹

商品证据由 `product-fingerprint.js` 生成，分三个等级：

| 等级 | 条件 | 分值 |
| --- | --- | ---: |
| `strong` | token 重叠率 ≥ 75% 且重叠 token 数 ≥ 2 | 40 |
| `weak` | token 重叠率 ≥ 50% 且重叠 token 数 ≥ 2 | 15 |
| `none` | 其余情况 | 0 |

工作原理：
1. 将访客 `product_id`（路径/URL）归一化为 handle → 拆分为 token 集合
2. 将订单 line_items 的标题清洗 → 拆分为 token 集合
3. 计算 token 重叠率（以较小集合为分母）

示例：
- 访客路径 `/products/rak-wastafel-dapur` → tokens: `[rak, wastafel, dapur]`
- 订单标题 `Rak Wastafel Dapur dengan Pintu (Not linked to Shopify)` → tokens: `[rak, wastafel, dapur, dengan, pintu]`
- 重叠 3/3 = 100% → **strong**

### 5.2 时间分

| 时间差 | 分值 |
| --- | ---: |
| ≤ 1h | 30 |
| > 1h 且 ≤ 6h | 20 |
| > 6h 且 ≤ 24h | 10 |
| > 24h | **淘汰** |

### 5.3 IP / Geo 证据

IP 证据由 `geo-evidence.js` 生成，分三层：

| 信号 | 条件 | 分值 |
| --- | --- | ---: |
| `browser_ip_exact` | 订单 browser_ip 与访客 IP 完全一致 | 25 |
| `browser_ip_mismatch` | browser_ip 存在但不一致 → 不再降级到 Geo | 0 |
| `geo_city` | 无 browser_ip 时，GeoIP 城市一致 | 10 |
| `geo_region` | 无 browser_ip 时，GeoIP 省/州一致 | 5 |
| `geo_country` | 无 browser_ip 时，GeoIP 国家一致 | **0**（仅记录，不加分） |

设计约束：
- geo_country **不加分**（按需求去掉）
- 有 browser_ip 但不一致时，不降级到 GeoIP
- Geo 证据不能单独放行

## 6. 决策层 — 主路径 / 降级路径

### 6.1 模式判定

每个候选根据商品证据等级分配模式：
- `productLevel === 'strong'` → **main（主路径）**
- 其余 → **fallback（降级路径）**

如果任何候选达到 main 模式，整体走主路径决策。

### 6.2 主路径

适用前提：存在至少一个 strong 商品证据候选

通过条件（同时满足）：
- 总分 ≥ **55**
- 第一名领先第二名 ≥ **10** 分

### 6.3 降级路径

适用前提：没有任何候选达到 strong 商品证据

通过条件（同时满足）：
- 总分 ≥ **35**
- 第一名领先第二名 ≥ **15** 分

### 6.4 拒绝原因码

| 原因码 | 含义 |
| --- | --- |
| `no_candidate` | 无候选（或候选列表为空） |
| `main_score_too_low` | 主路径最佳分数 < 55 |
| `main_gap_too_small` | 主路径领先分差 < 10 |
| `fallback_score_too_low` | 降级路径最佳分数 < 35 |
| `fallback_gap_too_small` | 降级路径领先分差 < 15 |

## 7. 访客占用与人工释放

### 7.1 占用模型

`matches` 表新增以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `active` | INTEGER DEFAULT 1 | 是否占用该访客 |
| `released_at` | TEXT | 人工撤销时间 |
| `released_reason` | TEXT DEFAULT '' | 撤销原因 |
| `match_mode` | TEXT DEFAULT '' | main / fallback |
| `lead_score_gap` | INTEGER DEFAULT 0 | 第一名与第二名分差 |
| `decision_summary` | TEXT DEFAULT '' | 成功匹配摘要 |

部分唯一索引：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_matches_active_visitor_unique
ON matches(visitor_id) WHERE active = 1;
```

这保证：
- 一个订单最多一条匹配记录
- 一个访客同一时刻只能被一个订单占用
- 人工释放后，该访客可重新参与匹配

### 7.2 人工撤销

接口：`POST /api/orders/:orderId/revoke-match`

行为：
1. 查订单和活跃匹配
2. 将 `active = 0`，写入 `released_at` 和 `released_reason`
3. 将订单状态改为 `matched_revoked`
4. 释放后的访客可被后续订单重新匹配

### 7.3 释放规则

- 回传失败**不会**自动释放
- 只有人工明确撤销时才释放
- 避免把"回传失败"误等同于"匹配错误"

## 8. 成功/失败摘要格式

### 8.1 成功摘要（decision_summary）

```
mode=main;product=strong;time_diff_minutes=35;ip=browser_ip_exact;lead_gap=12;click_source=tiktok
```

### 8.2 失败摘要（status_reason）

```
reason=fallback_gap_too_small;candidates=3;best_score=44;second_score=31;score_gap=13;best_time_diff_minutes=42
```

## 9. 订单状态流转

| 状态 | 含义 |
| --- | --- |
| `received` | 订单已写入但尚未处理 |
| `ignored_pending` | 财务状态 pending，跳过 |
| `duplicate_ignored` | 已有成功回传 |
| `unmatched` | 未匹配通过 |
| `matched_no_callback` | 匹配但未回传（平台凭证缺失等） |
| `callback_failed` | 匹配但回传失败 |
| `callback_sent` | 匹配并回传成功 |
| `matched_revoked` | 匹配被人工撤销 |
| `processing_failed` | 处理异常 |

## 10. 与 v1 机制的关键差异

| 维度 | v1 | v2 |
| --- | --- | --- |
| 时间窗 | 3 天（可配置） | 24 小时（硬编码） |
| 时间分满分 | 65 | 30 |
| 商品匹配 | 精确相等 | token 级模糊（强/弱/无） |
| 商品满分 | 25 | 40 |
| IP 满分 | 25 | 25 |
| geo_country 分 | 6 | 0（仅记录） |
| 最低阈值 | 25 | 55（主）/ 35（降级） |
| 歧义判定 | 分数 < 70 且分差 < 8 | 主路径分差 < 10 / 降级分差 < 15 |
| 访客复用 | 允许 | 禁止（同一时刻一个访客只能归属一个订单） |
| 人工释放 | 无 | 支持 |
| 匹配摘要 | 基础 status_reason | 结构化 decision_summary |
